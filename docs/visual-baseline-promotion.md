# Visual Baseline Promotion

Visual snapshots are operating-system-specific. Windows baselines are committed,
but Linux images must be generated by GitHub's Ubuntu runner rather than copied or
renamed from a developer machine.

## Promotion procedure

1. Run **Actions -> Visual Baselines (Linux) -> Run workflow**.
2. Leave `create_pr` enabled. The job regenerates Chromium and mobile-Chrome
   snapshots, checks that all six expected PNGs exist and have valid PNG/IHDR
   headers, and then reruns the visual suite without update mode.
3. Download the `visual-baselines-linux` artifact and review every image. Check
   text wrapping, focus indicators, clipping, and whether the masked canvases
   hide only simulation pixels.
4. Review and merge the workflow-created PR only if the visual change is
   intentional. Do not accept a large diff merely by increasing Playwright's
   pixel threshold.

Once any Linux snapshots are committed, mainline enforces a complete-set
contract with `npm run test:visual:linux:contract` and runs the Linux visual
gate. A partial promotion therefore fails rather than silently reducing
coverage. The workflow artifact remains available when `create_pr` is disabled,
but that mode does not activate the mainline gate.

macOS baselines remain unpromoted because they must be generated and reviewed on
a GitHub macOS runner. The same contract script supports them with
`node scripts/check-visual-baselines.mjs --platform=darwin`; a future macOS
promotion must add all six images atomically and a mainline macOS execution
lane before the repository claims macOS visual gating.
