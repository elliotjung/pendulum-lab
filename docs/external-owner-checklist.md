# External owner checklist

`lastVerified: 2026-07-13`

`staleAfterDays: 30`
`owner: repository maintainer unless a row names another owner`

This is the boundary between work the repository can automate and work that
needs an account, a physical device, a human review, or an irreversible publish
decision. A checked box means the public URL or evidence artifact was inspected,
not merely that a command exited successfully. Never put access tokens, runner
registration logs, private contact details, or unredacted device names in Git.

## Release identities

### npm - maintainer, external account required

- [ ] Sign in to npmjs.com as the owner of `@elliotjung` and create/confirm the
      package coordinate `@elliotjung/pendulum-lab` before the first publish.
- [ ] In the package's **Trusted Publisher** settings, bind GitHub user
      `elliotjung`, repository `pendulum-lab`, workflow `publish-npm.yml`,
      environment `npm`, and allowed action `npm publish`. The official npm guide
      is <https://docs.npmjs.com/trusted-publishers/>.
- [ ] Confirm the workflow uses a GitHub-hosted runner, `id-token: write`, Node
      24, npm >= 11.5.1, and an exact repository URL. These are pinned in
      `.github/workflows/publish-npm.yml`; do not add a long-lived npm token.
- [ ] Dispatch the workflow once with `dry_run=true` and the exact version, then
      approve the protected `npm` environment and run with `dry_run=false`.
- [ ] Evidence: the public npm version page resolves, its provenance points to
      this repository/workflow/tag, `npm view @elliotjung/pendulum-lab@<version>
dist.integrity` resolves, and `npm run release:status` reports the new scoped
      coordinate. The currently committed publication-status report predates the
      package rename and must be regenerated before release.

### JSR - maintainer, external account required

- [ ] Sign in at <https://jsr.io/new>, claim/confirm scope `@elliotjung`, create
      package `pendulum-lab`, and link it to `elliotjung/pendulum-lab` in package
      settings. JSR's official flow is
      <https://jsr.io/docs/publishing-packages>.
- [ ] Run `npm run jsr:check`; review every included source file and fix slow
      types or portability errors rather than bypassing them.
- [ ] Dispatch `.github/workflows/publish-jsr.yml` with `dry_run=true`. For the
      real release, ensure `jsr.json`, `package.json`, and tag versions are equal,
      then publish through GitHub OIDC.
- [ ] Evidence: `https://jsr.io/@elliotjung/pendulum-lab` resolves at the tag
      version, repository linkage is verified, generated docs expose the intended
      subpaths, and the provenance/source commit matches the GitHub release.

### Zenodo DOI - maintainer, irreversible publish

- [ ] Register or link the owner account and create a least-privilege token with
      `deposit:write` and `deposit:actions`; keep it only in the current process or
      an approved secret store. The official API/scopes are documented at
      <https://developers.zenodo.org/>.
- [ ] Test separately with a sandbox account/token:
      `$env:ZENODO_SANDBOX_TOKEN='<secret>'; npm run zenodo:draft -- --sandbox`.
      A sandbox DOI is not publication evidence.
- [ ] Review `.zenodo.json`, authorship, license, version, release tag, paper,
      reviewer manifest, and checksums. Then set `ZENODO_TOKEN` and run
      `npm run zenodo:publish` exactly once after owner approval.
- [ ] Run `npm run doi:sync`, inspect the `CITATION.cff`/README diff, rerun the
      release package, and commit the real DOI only.
- [ ] Evidence: `reports/zenodo-deposition.json` says published, the production
      `10.5281/zenodo.*` URL resolves, files/checksums match the release, and
      `npm run release:status` binds the DOI and tag to the same version.

### arXiv nlin.CD - author plus eligible endorser

- [ ] Create the submitter account early and start endorsement before the final
      release. Follow the official policy at
      <https://info.arxiv.org/help/endorsement.html>; do not ask strangers in bulk
      or treat endorsement as scientific review.
- [ ] Ask a teacher, supervisor, or known physicist who is eligible in the
      target category. Send a short abstract, author contribution statement,
      `paper/paper.pdf`, reviewer-kit link, and the endorsement code generated by
      arXiv. Record consent privately, not in the repository.
- [ ] After a Zenodo DOI exists, rebuild/review `paper/paper.pdf`; verify title,
      author identity, references, figure legibility, caveats, and the category
      choice `nlin.CD` (with `physics.comp-ph` only if appropriate). Use the current
      submission instructions at <https://info.arxiv.org/help/submit/>.
- [ ] Submit the PDF-only path already selected by the project. Complete arXiv's
      metadata/licensing screens manually and inspect the generated preview before
      finalizing.
- [ ] Evidence: a public arXiv identifier resolves after moderation; update
      `CITATION.cff`, preferred citation, landing scholarly metadata, and release
      docs, then regenerate the reviewer/release manifests.

## Deployment and communication

### Cloudflare Pages mirror - maintainer, external account and DNS optional

- [ ] Create a Cloudflare Pages project connected read-only to this repository
      or deploy the reviewed `dist/` artifact. Pin the production branch and build
      command; do not grant account-wide write access when project-scoped access is
      available.
- [ ] Add a `_headers` file in the deployed static output only after testing the
      complete app under `Cross-Origin-Opener-Policy: same-origin` and
      `Cross-Origin-Embedder-Policy: require-corp`. Cloudflare's official static
      header mechanism is <https://developers.cloudflare.com/pages/configuration/headers/>.
- [ ] Run smoke, reviewer-console, paper, worker, WASM, font, and third-party
      resource checks on the `pages.dev` URL. `crossOriginIsolated === true` is
      necessary for SharedArrayBuffer work but is not by itself proof of correctness.
- [ ] Keep GitHub Pages as the stable fallback until the mirror passes the same
      release tag. A custom domain/DNS cutover is a separate owner decision.
- [ ] Evidence: public mirror URL, deployment commit, response-header capture,
      Playwright report, rollback target, and a note showing no asset was blocked by
      COEP.

### Narrated demo video - generated locally; external hosting optional

The repository ships `reports/demo-narrated-ko.mp4` with synchronized WebVTT
captions and a transcript. The release workflow uploads all three, so a YouTube
account is optional rather than a publication blocker.

- [ ] Record a 60-120 second script: question -> reproduce command -> Figure 1
      gap map -> Trust Inspector/caveat -> reviewer console -> standalone download.
      Do not claim NVIDIA/AMD, DOI, npm, or arXiv completion unless the public
      evidence above exists at recording time.
- [ ] Capture at 1080p with readable 125% UI scale, no notifications/account
      identifiers, and captions. Include spoken and on-screen explanations of
      `A_c`, `A_PD`, and why their crossing is not "Melnikov predicts the cascade."
- [ ] Export a captioned MP4 plus WebVTT/transcript. A YouTube upload may be
      unlisted until reviewed; the landing page should use a self-hosted thumbnail
      link rather than an embed unless CSP/privacy policy is intentionally changed.
- [ ] Evidence: final video URL or release asset, transcript, duration, source
      release tag, thumbnail alt text, and a two-person factual/accessibility review.

## Human and physical validation

### NVDA screen-reader scenario - Windows tester

- [ ] Install the current NVDA release from the official NV Access page:
      <https://www.nvaccess.org/download/>. Record NVDA, Windows, browser, and app
      release versions; do not record the tester's name without consent.
- [ ] With the mouse set aside, open the app, reach the simulator, start/pause a
      run, change one parameter, reach Validation/Trust Inspector, and read the
      reported Lyapunov value and trust/caveat text.
- [ ] Check focus order, visible focus, control name/role/state, live update
      interruption, canvas alternative text/data, keyboard trap absence, and
      recovery after closing dialogs. Test both browse and focus modes where
      relevant.
- [ ] Log exact steps, expected/actual speech, severity, workaround, and issue
      link in a dated `docs/accessibility-manual-validation.md` entry. Never write
      "screen-reader verified" from axe results alone.

### Mid-range Android field profile - device owner

- [ ] Choose a physical mid-range Android phone representative of student use;
      record model class, SoC/RAM, Android, Chrome, display resolution/refresh rate,
      battery/thermal state, and network condition. Redact device serials and owner
      identifiers.
- [ ] Test cold load, warm load, first interaction, 60-second balanced run,
      quality downgrade/recovery, long trail/density view, orientation change,
      background/foreground, and reviewer popover. Repeat at least three times and
      report median plus range, not the best run.
- [ ] Capture `performance.memory` only when supported; otherwise use trace,
      frame timing, long-task count, dropped-frame observation, battery/thermal
      notes, and screenshot/video evidence. Compare against `docs/performance.md`
      budgets without inventing unsupported metrics.
- [ ] Evidence: anonymized raw trace, environment JSON, summary table, release
      commit, observed limitations, and follow-up issue links committed under an
      agreed reports path.

### NVIDIA/AMD guest evidence - maintainer plus device owner

- [ ] Follow the registration, isolation, artifact acceptance, and mandatory
      revocation procedure in `docs/hardware-validation.md`.
- [ ] Evidence is complete only when the artifact hash is recorded and the
      ephemeral runner and temporary OS account are both removed.

## Staleness rule

At each release, compare today's date to `lastVerified`. When older than
`staleAfterDays`, the maintainer must reopen every official link, inspect the
current workflow/account UI, update changed requirements, and set a new
`lastVerified`. A broken or materially changed source makes the affected lane
`blocked-external`; it must not be silently checked off from an older release.
